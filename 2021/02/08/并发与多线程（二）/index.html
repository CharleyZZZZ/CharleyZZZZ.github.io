Java线程池

创建线程池

线程池通过复用线程，避免线程频繁地创建和销毁。Java 的 Executors 工具类中提供了 5 种类型的线程池创建方法：

  • newFixedThreadPool：固定线程数，无界队列，适用于任务数量不均匀的场景，对内存压力不敏感，但系统负载比较敏感的场景；核心和最大线程数都设置成指定的线程数。队列使用无界队列 LinkedBlockingQueue。
  • newCachedThreadPool：无限线程数，适用于要求低延迟的短期任务场景；核心线程数设置为 0，最大线程数是 Integer.MAX_VALUE，主要是通过把缓冲队列设置成 SynchronousQueue，这样只要没有空闲线程就会新建。
  • newSingleThreadExecutor：单个线程的固定线程池，适用于保证异步执行顺序的场景；线程数设置为 1 的固定线程池。
  • newScheduledThreadPool：支持固定频率和固定延迟，适用于定期执行任务的场景；队列使用DelayedWorkQueue，这是一种按延迟时间获取任务的优先级队列。
  • newWorkStealingPool：使用ForkJoinPool，多任务队列的固定并行度，适合任务执行时长不均匀的场景。

以上除了 工作窃取线程池 (newWorkStealingPool )外,其他都是通过 ThreadPoolExecutor 的不同初始化参数来创建的。

线程池任务执行流程

向线程提交任务时可以使用 execute 和 submit

  • submit 可以返回一个 future 对象，通过 future 对象可以了解任务执行情况，可以取消任务的执行，还可获取执行结果或执行异常。
  • submit 最终也是通过 execute 执行的。

ThreadPoolExecutor类

ThreadPoolExecutor 继承了AbstractExecutorService类，AbstractExecutorService是一个抽象类，它实现了ExecutorService接口。而ExecutorService又是继承了Executor接口，我们看一下Executor接口的代码：

Executor是一个顶层接口，在它里面只声明了一个方法execute(Runnable)，返回值为void，参数为Runnable类型，从字面意思可以理解，就是用来执行传进去的任务的；

然后ExecutorService接口继承了Executor接口，并声明了一些方法：submit、invokeAll、invokeAny以及shutDown等；

抽象类AbstractExecutorService实现了ExecutorService接口，基本实现了ExecutorService中声明的所有方法；

然后ThreadPoolExecutor继承了类AbstractExecutorService。

ThreadPoolExecutor 提供俩个方法用于线程池的关闭：

  • shutdown()：不会立即终止线程池，而是要等所有任务缓存队列中的任务都执行完后才终止，但再也不会接受新的任务
  • shutdownNow()：立即终止线程池，并尝试打断正在执行的任务，并且清空任务缓存队列，返回尚未执行的任务

线程池大小设置多少合适？

最佳线程数目 = （（线程等待时间+线程CPU时间）/线程CPU时间 ）* CPU数目

这里 线程等待时间 可以理解为 线程IO时间，线程CPU时间 可以由 （请求时间 - 线程IO 时间）粗略算得。

线程安全 - 并发容器

ArrayList -> CopyOnWriteArrayList

写入时复制（CopyOnWrite）思想

　　写入时复制（CopyOnWrite，简称COW）思想是计算机程序设计领域中的一种优化策略。其核心思想是，如果有多个调用者（Callers）同时要求相同的资源（如内存或者是磁盘上的数据存储），他们会共同获取相同的指针指向相同的资源，直到某个调用者试图修改资源内容时，系统才会真正复制一份专用副本（private copy）给该调用者，而其他调用者所见到的最初的资源仍然保持不变。这过程对其他的调用者都是透明的。

此做法主要的优点是如果调用者没有修改资源，就不会有副本（private copy）被创建，因此多个调用者只是读取操作时可以共享同一份资源。

CopyOnWriteArrayList源码解析

重点关注其 add(E e) 方法，可以看到 其方法内部 使用了 锁，每次新增元素 都是 Arrays.copyOf 方法新建一个对象数组，长度为（len + 1）,然后 替换 原数组，所以新增元素开销大。

  • CopyOnWriteArrayList 实现了List接口。
  • 内部持有 一个 ReentrantLock 锁对象，新增元素时 加锁。
  • 底层是用volatile transient声明的数组 array。（volatile 保证可见性，强制主存刷新，防止指令重排，transient 防止序列化）
  • 读写分离，写时复制出一个新的数组，完成插入、修改或者移除操作后将新数组赋值给array。
  • 可以看到 每次 新增都需要复制新建 数组，开销大，所以尽量使用 批量添加，提升性能。
  • 与Vector 比较，Vector是增删改查方法都加了synchronized，保证同步，但是每个方法执行的时候都要去获得锁，性能就会大大下降，而CopyOnWriteArrayList 只是在增删改上加锁，但是读不加锁，在读方面的性能就好于Vector，CopyOnWriteArrayList支持读多写少的并发情况。
  • CopyOnWrite容器只能保证数据的最终一致性，不能保证数据的实时一致性。所以如果你希望写入的的数据，马上能读到，请不要使用CopyOnWrite容器。

同理 TreeSet -> CopyOnWriteArraySet 不再赘述。也可自己实现 CopyOnWriteHashMap试试。

总结：

  • ConcurrentHashMap 1.8实现方式 类似于 HashMap,底层是由 数组 +链表（红黑树）方式，只是添加了 CAS + synchronized 保证线程安全。
  • ConcurrentHashMap 链表数量达到阈值后，并未直接转换为 红黑树，而是先判断 数组长度是否 小于64 ，如果小于64，则优先 扩容数组。否则转换为红黑树。
  • ConcurrentHashMap 扩容时，空过 sizeCtl 控制多线程一起扩容，即当ConcurrentHashMap 扩容时，其他线程并非空闲等待，而是帮助扩容线程共同完成扩容操作。
    • sizeCtl 为 -1 时，ConcurrentHashMap正在初始化
    • sizeCtl 为 正数时，属于使用阶段，sizeCtl = size * 0.75 （阈值）
    • sizeCtl 为 负数时，处于扩容阶段，-N ,其中 -N 的低16位二进制 = 正在参与扩容线程数 + 1

与 jdk1.8 实现方式不同的是，jdk1.7 使用了锁分段技术，首先将数据分成一段一顿的存储,然后给每一段数据配一把锁,当一个线程占用锁访问其中一段数据的时候,其他段的数据也能被其他线程访问到.另外ConcurrentHashMap可以做到读取数据不加锁。

ConcurrentHashMap是由Segment数组结构和HashEntry数组结构组成.Segment是一种可重用锁ReentrantLock, HashEntry用于存储键值对数据.一个ConcurrentHashMap中包含一个Segement数组,

每个 Segement 结构和 HashMap 类似,是一种数组和链表结构.一个 Segment 里包含一个 HashEntry 数组,每一个 HashEntry 是一个链表结构,每当对 HashEntry 数组的数据进行修改时,必须首先获得它对应的 Segment 锁.

ConcurrentHashMap查询 定位一个元素需要进行俩次Hash操作，第一次 Hash 定位到segment，第二次Hash定位到 链表头部，所以Hash过程比一般HashMap要久。

ConcurrentHashMap 写操作只需要对 segment 加锁即可，不会影响其他segment 操作，所以理想情况下，并发数为 segment 数量。

还需要注意的一点是，1.7 中 size 是需要统计 所有segment 内元素数量相加，需要所有segment 加锁统计，但是 因为性能问题作者并未 一开始 就全部加锁，而是采用俩次统计的方式，如果俩次统计对比数量一致，则期间并未涉及元素增删，如果俩次统计数量不一致，则再全部segment加锁统计。

补充：HashTable 和 Collections.synchronizeMap() (内部实现为 SynchronizeMap) 都是使用synchronized 关键字，性能相比ConcurrentHashMap 较低一些。

AQS（AbstractQueuedSynchronizer）

抽象的队列式的同步器（AbstractQueuedSynchronizer），AQS定义了一套多线程访问共享资源的同步器框架，许多同步类实现都依赖于它，如常用的ReentrantLock/Semaphore/CountDownLatch等。

特点：

  1. 使用Node实现FIFO队列，可以用于构建锁或则其他同步装置的基础框架
  2. 利用一个int类型（state）表示状态
  3. 使用方法是基础
  4. 子类通过继承并通过实现它的方法管理其状态 { acquire 和 release} 的方法操纵状态
  5. 可以同时实现排他锁和共享锁模式（独占、共享）

AQS维护了一个 volatile int state（代表共享资源）和一个FIFO线程等待队列（多线程争用资源被阻塞时会进入此队列）。这里volatile是核心关键词。state的访问方式有三种:

  • getState()
  • setState()
  • compareAndSetState()

AQS定义两种资源共享方式：

  • Exclusive（独占，只有一个线程能执行，如ReentrantLock）
  • Share（共享，多个线程可同时执行，如Semaphore/CountDownLatch）

不同的自定义同步器争用共享资源的方式也不同。自定义同步器在实现时只需要实现共享资源state的获取与释放方式即可，至于具体线程等待队列的维护（如获取资源失败入队/唤醒出队等），AQS已经在顶层实现好了。自定义同步器实现时主要实现以下几种方法：

  • isHeldExclusively()：该线程是否正在独占资源。只有用到condition才需要去实现它。
  • tryAcquire(int)：独占方式。尝试获取资源，成功则返回true，失败则返回false。
  • tryRelease(int)：独占方式。尝试释放资源，成功则返回true，失败则返回false。
  • tryAcquireShared(int)：共享方式。尝试获取资源。负数表示失败；0表示成功，但没有剩余可用资源；正数表示成功，且有剩余资源。
  • tryReleaseShared(int)：共享方式。尝试释放资源，如果释放后允许唤醒后续等待结点返回true，否则返回false。
